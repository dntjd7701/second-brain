# 성과와 기술 성장 포트폴리오

이 문서는 내가 어떤 일을 해왔고, 어떤 기술적 성장을 했고, 어떤 방식으로 문제를 해결하는 사람인지 설명하기 위한 누적 포트폴리오다.

이력서는 이 문서의 목적이 아니라 결과물 중 하나다. 이 문서에 축적된 내용을 바탕으로 이력서, 자기소개, 면접 답변, 성과 리뷰에 필요한 표현을 나중에 선별한다.

상세 분석은 각 기술 문서나 실패 기록에 작성하고, 이 문서에는 핵심 요약과 링크만 남긴다.

## 작성 기준

각 항목은 짧게 요약하되 다음 질문에 답해야 한다.

- 왜 중요한 문제였는가?
- 내가 어떤 역할을 했는가?
- 왜 이 기술 또는 접근 방식을 선택했는가?
- 어떤 대안을 검토했고 왜 제외했는가?
- 어떻게 개선했는가?
- 어떤 트레이드오프가 있었는가?
- 어떤 기대효과가 있는가?
- 기술적으로 어떤 가치가 있는가?
- 비즈니스 또는 운영 관점에서 어떤 가치가 있는가?
- 이 일을 통해 어떤 기술적 성장을 했는가?
- 다른 사람에게 설명 가능한 성과 서사는 무엇인가?
- 정량화 가능한 지표가 있는가?
- 상세 내용은 어느 문서에서 볼 수 있는가?

## 항목 상태

- `raw`: 의미 있는 작업으로 기록했지만 아직 정리가 더 필요함
- `refined`: 설명 가능한 성과 서사로 정제됨
- `resume-ready`: 이력서나 자기소개에 바로 활용 가능함
- `reflected`: `resume/master-resume.md` 등 파생 문서에 반영됨

## EMR 포틀릿 환자 변경 안정성

- 날짜: 2026-05-18
- 프로젝트: Amaranth10 EMR Front
- 한 줄 요약: EMR 진료 메인에서 환자 전환 시 미저장 데이터가 유실될 수 있는 경로를 분석하고, 포틀릿 공통 dirty 집계 구조를 활용해 환자 변경 전 확인 절차를 추가했다.
- 내가 맡은 역할: `PatientInfo`, `PrscVisitHistoryDialog`, `PatientInfoWidget`, `Portlet` dirty 집계 구조를 추적해 환자 변경 경로와 누락된 guard 지점을 분석했다.
- 왜 중요했는가: 환자 컨텍스트 변경은 임상 업무에서 데이터 유실과 오입력 리스크로 이어질 수 있어 운영 안정성과 사용자 신뢰에 직접적인 영향을 준다.
- 어떻게 개선했는가: `PatientInfoWidget.onPatientChange`에서 `portlet.hasDirty()`를 호출하고, dirty가 있으면 `ConfirmMessageEnum.clearDirty` 확인 후에만 `portlet.setState({ patientInfo })`를 실행하도록 했다.
- 기대효과: 내원이력 기반 환자 전환 시 미저장 처방/상병/메모가 의도치 않게 사라지는 상황을 줄일 수 있다.
- 기술적 성장: 단일 컴포넌트 수정이 아니라 포틀릿 컨텍스트, 환자 정보 위젯, 내원이력 다이얼로그 사이의 비동기 상태 변경 계약을 추적해 안전한 변경 지점을 판단했다.
- 설명 가능한 성과 서사: EMR 진료 메인에서 환자 전환 시 미저장 데이터가 유실될 수 있는 경로를 찾아, 기존 포틀릿 dirty 집계 구조를 활용해 환자 변경 전에 사용자 확인을 거치도록 개선했다.
- 상세 문서: [[technical-notes/EMR 포틀릿 환자 변경 안정성]]
- 사용 기술: React, luna-orbit, klago-ui-hospital-common, Portlet Context
- 이력서/면접 활용 문장 초안: EMR 진료 메인 포틀릿의 환자 컨텍스트 변경 경로를 분석하고, 공통 dirty 집계 구조를 활용한 미저장 데이터 보호 로직을 적용해 환자 전환 시 데이터 유실 리스크를 줄였다.
- 정량화 가능 지표: 환자 전환 경로 수, dirty confirm 발생 케이스, 관련 VOC/버그 감소 여부를 추적하면 정량화 가능
- 상태: raw

## Voice Note 기반 Voice to EMR 경과기록 PoC

- 날짜: 2026-06-12
- 프로젝트: Amaranth10 EMR Front / Medical Backend
- 한 줄 요약: 진료 대화 녹음, STT, 화자 역할 분류, SOAP 경과기록 초안 생성, 기존 저장 흐름 연결까지 Voice Note 기반 경과기록 작성 보조 흐름을 설계하고 구현했다.
- 내가 맡은 역할: MEDA0010 Voice Note UI, 녹음/중지/재시작 흐름, 전사 전문 검토, SOAP 재생성, prompt/schema 계약, OpenAI Responses API 연동, backend prompt resource 로딩 정책을 함께 조율했다.
- 왜 중요했는가: 의료 문서 작성 업무에서 AI 결과가 곧바로 저장되는 위험을 피하면서도, 실제 진료 흐름 안에 음성 기반 초안 작성 경험을 자연스럽게 넣는 것이 핵심이었다.
- 어떻게 개선했는가: 녹음 중에는 파형과 상태 표시로 사용자 피드백을 제공하고, 녹음 종료 후 STT와 SOAP 정규화를 거쳐 사용자가 검토/수정한 뒤 기존 경과기록 저장 흐름을 타도록 책임 경계를 분리했다.
- 기대효과: 경과기록 입력 부담을 줄이고, 의료진이 AI 생성 내용을 직접 검토한 뒤 저장하도록 하여 문서 작성 속도와 안전성을 함께 확보할 수 있다.
- 기술적 성장: 프론트 UI 상태 관리, MediaRecorder 기반 녹음 UX, backend RestTemplate 기반 AI gateway, JSON schema structured output, prompt fail-fast 정책을 하나의 업무 흐름으로 연결했다.
- 설명 가능한 성과 서사: Voice Note 기반 경과기록 초안 작성 PoC를 설계하고 구현하면서, AI 결과를 직접 저장하지 않고 사용자 검토와 기존 저장 프로세스를 유지하는 안전한 Voice to EMR 업무 흐름을 만들었다.
- 상세 문서: [[technical-notes/Voice to EMR 경과기록지 1차 PoC 의사결정]], [[process/Voice to EMR 경과기록지 구현 절차]]
- 사용 기술: React, MediaRecorder, Web Audio API, Java Spring, RestTemplate, OpenAI Responses API, JSON Schema, prompt engineering
- 이력서/면접 활용 문장 초안: 의료진의 경과기록 작성 부담을 줄이기 위해 Voice Note 기반 STT/SOAP 초안 생성 PoC를 구현하고, AI 결과가 기존 저장 흐름을 우회하지 않도록 사용자 검토 중심의 안전한 EMR 입력 구조를 설계했다.
- 정량화 가능 지표: 녹음 후 초안 생성 소요 시간, SOAP 적용률, 사용자 수정률, 재녹음률, 경과기록 작성 시간 감소 여부를 추적하면 정량화 가능
- 상태: raw

## 감염병 신고 외부 연계 오류 처리 안정화

- 날짜: 2026-06-12
- 프로젝트: Amaranth10 EMR 감염병 신고
- 한 줄 요약: 감염병 신고/고지 외부 API 연계에서 실패 원인과 사용자 메시지가 흐려지는 경로를 추적하고, FE/BE 요청 흐름과 오류 표시 책임을 정리했다.
- 내가 맡은 역할: 감염병 신고 다이얼로그의 form state, backend infection service route, 외부 API timeout/400 응답, FE alert/message 표시 경로를 실제 요청 흐름 기준으로 분석했다.
- 왜 중요했는가: 감염병 신고는 외부 기관 연계와 법정 신고 업무에 닿아 있어, 실패 시 원인을 사용자가 인지하고 재시도/문의할 수 있게 만드는 것이 운영 안정성에 직접 연결된다.
- 어떻게 개선했는가: notice/report 경로를 분리해 추적하고, 외부 API 실패가 generic fallback으로 덮이지 않도록 오류 발생 위치와 사용자 표시 경로를 명확히 했다.
- 기대효과: 외부 연계 실패 시 원인 파악 시간이 줄고, 신고 업무 사용자가 막연한 실패 메시지 대신 조치 가능한 오류를 확인할 수 있다.
- 기술적 성장: FE form state와 backend RestTemplate 외부 연계, 예외 변환, 사용자 메시지 표시를 하나의 장애 흐름으로 추적하는 디버깅 역량을 강화했다.
- 설명 가능한 성과 서사: 감염병 신고 외부 연계 오류 흐름을 FE 다이얼로그부터 backend infection service까지 추적해, 실패 원인이 사용자에게 명확히 전달되도록 오류 처리와 메시지 표시 구조를 안정화했다.
- 상세 문서: 추후 감염병 신고 오류 처리 기술 노트로 분리 후보
- 사용 기술: React, Java Spring, RestTemplate, external API integration, error handling
- 이력서/면접 활용 문장 초안: 감염병 신고 외부 API 연계 실패 흐름을 프론트 상태와 백엔드 예외 처리까지 추적해, 운영자가 원인을 파악할 수 있는 오류 표시 구조로 개선했다.
- 정량화 가능 지표: 신고 실패 문의 건수, timeout/400 응답 재현 시간, 오류 메시지 식별률, 관련 VOC 감소 여부를 추적하면 정량화 가능
- 상태: raw

- 날짜: 2026-06-12
- 프로젝트: Amaranth10 의료쇼핑방지망
- 한 줄 요약: 의료쇼핑방지망 연계에 필요한 병원별 인증/암호화 설정을 FE 상수나 범용 설정 조회가 아니라 BE 전용 병원환경설정 조회 흐름으로 정리했다.
- 내가 맡은 역할: 의쇼망 공개키/기관 설정/팝업 URL/저장 전 검증 흐름을 FE와 BE 책임으로 나누고, 민감 설정값이 FE나 문서에 노출되지 않는 구조를 검토했다.
- 왜 중요했는가: 의료쇼핑방지망은 병원별 가입 정보와 암호화 키가 필요하므로, 코드 상수나 FE 노출 구조로 처리하면 운영 대응성과 보안 책임이 약해진다.
- 어떻게 개선했는가: FE는 필요한 업무 요청만 전달하고, BE가 세션 기준 병원환경설정과 환자정보를 조합해 외부 API 호출과 검증 결과를 책임지는 구조로 정리했다.
- 기대효과: 병원별 설정 변경에 배포 없이 대응하기 쉬워지고, 민감 설정값 노출면을 줄이며, 저장 전 검증의 책임 위치가 명확해진다.
- 기술적 성장: 외부 연계 설정값을 UI 편의 기준이 아니라 보안/운영 책임 기준으로 분리하는 backend-owned integration 패턴을 정리했다.
- 설명 가능한 성과 서사: 의료쇼핑방지망 연계에서 병원별 공개키와 인증 설정을 백엔드 병원환경설정 기반으로 처리하도록 정리해, 보안성과 운영 설정 대응력을 높였다.
- 상세 문서: [[technical-notes/의료쇼핑방지망 병원환경설정 키 연계]]
- 사용 기술: React, Java Spring, hospital system configuration, RSA public key usage, external API integration
- 이력서/면접 활용 문장 초안: 의료쇼핑방지망 연계에 필요한 병원별 인증/암호화 설정을 백엔드 병원환경설정 기반으로 분리해, 민감 설정값 노출을 줄이고 운영 환경별 대응력을 높였다.

- 한 줄 요약: 외부 원천 데이터를 표준 진료 데이터로 이관하기 위한 실행 프로그램, 전용 프로시저, 임시 적재/검증/최종 반영 흐름을 정리했다.
