# 성과와 기술 성장 포트폴리오

이 문서는 내가 어떤 일을 해왔고, 어떤 기술적 성장을 했고, 어떤 방식으로 문제를 해결하는 사람인지 설명하기 위한 누적 포트폴리오다.

이력서는 이 문서의 목적이 아니라 결과물 중 하나다. 이 문서에 축적된 내용을 바탕으로 이력서, 자기소개, 면접 답변, 성과 리뷰에 필요한 표현을 나중에 선별한다.

상세 분석은 각 기술 문서나 실패 기록에 작성하고, 이 문서에는 핵심 요약과 링크만 남긴다.

## 작성 기준

각 항목은 짧게 요약하되 다음 질문에 답해야 한다.

- 왜 중요한 문제였는가?
- 내가 어떤 역할을 했는가?
- 왜 이 기술 또는 접근 방식을 선택했는가?
- 어떤 대안을 검토했고 왜 제외했는가?
- 어떻게 개선했는가?
- 어떤 트레이드오프가 있었는가?
- 어떤 기대효과가 있는가?
- 기술적으로 어떤 가치가 있는가?
- 비즈니스 또는 운영 관점에서 어떤 가치가 있는가?
- 이 일을 통해 어떤 기술적 성장을 했는가?
- 다른 사람에게 설명 가능한 성과 서사는 무엇인가?
- 정량화 가능한 지표가 있는가?
- 상세 내용은 어느 문서에서 볼 수 있는가?

## 항목 상태

- `raw`: 의미 있는 작업으로 기록했지만 아직 정리가 더 필요함
- `refined`: 설명 가능한 성과 서사로 정제됨
- `resume-ready`: 이력서나 자기소개에 바로 활용 가능함
- `reflected`: `resume/master-resume.md` 등 파생 문서에 반영됨

## EMR 포틀릿 환자 변경 안정성

- 날짜: 2026-05-18
- 프로젝트: Amaranth10 EMR Front
- 한 줄 요약: EMR 진료 메인에서 환자 전환 시 미저장 데이터가 유실될 수 있는 경로를 분석하고, 포틀릿 공통 dirty 집계 구조를 활용해 환자 변경 전 확인 절차를 추가했다.
- 내가 맡은 역할: `PatientInfo`, `PrscVisitHistoryDialog`, `PatientInfoWidget`, `Portlet` dirty 집계 구조를 추적해 환자 변경 경로와 누락된 guard 지점을 분석했다.
- 왜 중요했는가: 환자 컨텍스트 변경은 임상 업무에서 데이터 유실과 오입력 리스크로 이어질 수 있어 운영 안정성과 사용자 신뢰에 직접적인 영향을 준다.
- 어떻게 개선했는가: `PatientInfoWidget.onPatientChange`에서 `portlet.hasDirty()`를 호출하고, dirty가 있으면 `ConfirmMessageEnum.clearDirty` 확인 후에만 `portlet.setState({ patientInfo })`를 실행하도록 했다.
- 기대효과: 내원이력 기반 환자 전환 시 미저장 처방/상병/메모가 의도치 않게 사라지는 상황을 줄일 수 있다.
- 기술적 성장: 단일 컴포넌트 수정이 아니라 포틀릿 컨텍스트, 환자 정보 위젯, 내원이력 다이얼로그 사이의 비동기 상태 변경 계약을 추적해 안전한 변경 지점을 판단했다.
- 설명 가능한 성과 서사: EMR 진료 메인에서 환자 전환 시 미저장 데이터가 유실될 수 있는 경로를 찾아, 기존 포틀릿 dirty 집계 구조를 활용해 환자 변경 전에 사용자 확인을 거치도록 개선했다.
- 상세 문서: [[technical-notes/EMR 포틀릿 환자 변경 안정성]]
- 사용 기술: React, luna-orbit, klago-ui-hospital-common, Portlet Context
- 이력서/면접 활용 문장 초안: EMR 진료 메인 포틀릿의 환자 컨텍스트 변경 경로를 분석하고, 공통 dirty 집계 구조를 활용한 미저장 데이터 보호 로직을 적용해 환자 전환 시 데이터 유실 리스크를 줄였다.
- 정량화 가능 지표: 환자 전환 경로 수, dirty confirm 발생 케이스, 관련 VOC/버그 감소 여부를 추적하면 정량화 가능
- 상태: raw
