# Second Brain Vault System

## 역할

기록할 가치가 있는 작업을 Obsidian Vault의 장기 기억으로 연결한다.

이 스킬은 항상 실행되는 코어가 아니다. second-brain-core가 기록 가치가 있다고 판단했거나, 사용자가 기록/문서화/Vault 반영을 요청했을 때 사용한다.

## 사용 조건

다음 경우에만 사용한다.

- 중요한 기술 결정, 아키텍처 판단, 도구 선택이 생겼다.
- 실패 원인, 잘못된 추측, 재발 방지 전략을 남길 가치가 있다.
- 반복 가능한 해결 방식이나 운영 절차가 생겼다.
- 업무, 기술, 비즈니스, 성장 관점에서 다시 읽을 가치가 있다.
- 기존 문서와 연결하거나 기존 문서를 확장하는 편이 자연스럽다.
- 사용자가 명시적으로 기록, 문서화, Vault 저장을 요청했다.

단순 작업 로그, 사소한 수정, 일회성 명령 결과는 문서화하지 않는다.

## Vault root 안전 규칙

Vault root는 절대 추정하지 않는다.

다음은 Vault root의 근거가 아니다.

- 현재 작업 디렉터리
- AGENTS.md가 있는 디렉터리
- README, PROJECTS, Personal, 사용자명처럼 보이는 경로
- 과거 세션에서 본 것 같은 기억
- 사용자가 단순히 작업 위치로 제공한 경로

사용자가 명시적으로 Vault root라고 말했거나, 이미 확정된 설정 문서가 있을 때만 Vault root로 취급한다.

Vault root가 불명확하면 파일 생성/수정/삭제를 멈추고 사용자에게 정확한 경로를 묻는다.

## 작업 흐름

1. 기록 가치가 있는지 먼저 판단한다.
2. 기존 Vault 문서와 연결 가능한지 찾는다.
3. 새 문서보다 기존 문서 확장을 우선한다.
4. 새 문서가 필요하면 지식의 성격을 먼저 고른다.
5. 문서는 작업 로그가 아니라 재사용 가능한 지식으로 작성한다.
6. 관련 문서가 있으면 Obsidian 링크로 연결한다.
7. 커리어 성과로 설명 가능하면 별도 요약 후보를 남긴다.

## Git publish 규칙

Vault 문서를 생성하거나 수정했다면 작성이 끝난 뒤 해당 Vault git 저장소에서 변경사항을 push한다.

- `git status`로 변경 범위를 확인한다.
- 이번 작업에서 생성하거나 수정한 Vault 문서만 stage한다. 기존에 있던 unrelated 변경은 stage하지 않는다.
- 커밋 메시지는 문서 주제를 알 수 있게 간결하게 작성한다.
- 커밋 후 현재 브랜치를 원격에 push한다.
- push가 실패하면 실패 원인과 남은 로컬 commit 여부를 사용자에게 보고한다.
- 민감정보, API key, 운영 DB 접속 정보, 환자정보가 포함된 문서는 stage/push하지 않는다.

## 지식 분류

- 결정: 장기 영향이 있는 설계, 정책, 도구 선택
- 실패: 재발 가능성이 있고 디버깅 가치가 있는 문제
- 패턴: 다음 작업에도 반복 적용할 수 있는 해결 방식
- 프로세스: 리뷰, 배포, 검증, 문서화처럼 반복 가능한 절차
- 기술 노트: 특정 기술, API, 상태 흐름, 아키텍처 설명
- 커리어 이벤트: 역할, 판단, 성과, 성장으로 설명 가능한 경험

하나의 작업이 여러 성격을 가지면 중심 주제 하나를 고르고, 나머지는 문서 안에서 링크로 연결한다.

## 작성 원칙

- 한국어로 작성한다. 코드 식별자, 파일 경로, API, 에러 메시지는 원문을 유지한다.
- 날짜나 티켓명보다 나중에 다시 찾을 수 있는 큰 주제명으로 제목을 정한다.
- 무엇을 했는지보다 왜 필요했고, 어떻게 판단했고, 다음에 어떻게 재사용할지를 쓴다.
- 기술적 가치와 비즈니스/운영 가치를 구분한다.
- 추정과 확인된 사실을 분리한다.
- 문서가 자연스럽게 읽히는 흐름을 해치면서까지 템플릿을 모두 채우지 않는다.

## 가벼운 템플릿

### Decision

- 배경
- 선택지
- 결정
- 버린 선택지와 이유
- 기대효과
- 장기 영향과 재검토 조건

### Failure

- 증상
- 잘못된 추측
- 실제 원인
- 디버깅 과정
- 해결
- 재발 방지 전략

### Pattern

- 반복되는 문제
- 적용 조건
- 해결 구조
- 예외와 주의사항
- 다음에 재사용할 기준

### Career Event

- 문제
- 내가 맡은 역할
- 어떻게 해결했는가
- 기술적 가치
- 비즈니스/운영 가치
- 설명 가능한 성과 문장 초안

## 보안 금지

다음 정보는 Vault에 저장하지 않는다.

- 비밀번호
- API Key
- Access Token / Refresh Token
- 운영 DB 접속 정보
- 민감 개인정보
- 보안 정보
