# Deep Research

## 목적
- 사용자의 요청을 바로 구현하지 않고, 요구사항이 충분히 정확해질 때까지 반복적으로 확인한다.
- 사용자의 답변을 반영해 요구사항 요약을 갱신하고, 사용자가 동의할 때까지 확정 게이트를 유지한다.
- 요구사항이 확정된 뒤에만 작업 분해, 실행 계획, 검증 방향으로 넘어간다.

## 언제 사용하나
- 요구사항이 크거나 모호할 때
- 사용자가 원하는 결과, 범위, 제외 범위, 성공 기준이 아직 불명확할 때
- 구현 전에 사용자와 요구사항을 합의해야 할 때
- 잘못된 가정으로 구현이 시작될 위험이 클 때

## 핵심 원칙
- 한국어로 작성한다.
- 요구사항이 확정되기 전에는 구현 계획이나 코드 수정으로 넘어가지 않는다.
- 확인 가능한 사실, 사용자 결정, 추정을 분리한다.
- 사용자가 이미 명확히 말한 내용은 다시 질문하지 않는다.
- 한 번에 너무 많은 질문을 하지 말고, 진행을 막는 핵심 질문 1~3개만 묻는다.
- 사용자의 답변을 반영해 요구사항 요약을 다시 작성한다.
- 사용자가 요약에 동의할 때까지 질문과 요약 갱신을 반복한다.
- 사용자가 명시적으로 `가정하고 진행`, `바로 진행`, `질문 없이 진행`을 요구하면 가정을 표시하고 다음 단계로 넘어갈 수 있다.

## 반복 절차
1. 원 요청을 한 문단으로 재정의한다.
2. 현재 확정된 내용, 모호한 내용, 추정한 내용을 분리한다.
3. 진행을 막는 핵심 질문을 1~3개만 묻는다.
4. 사용자 답변을 반영해 요구사항 요약을 갱신한다.
5. 남은 모호점이 있는지 다시 판단한다.
6. 요구사항 요약을 사용자에게 보여주고 확정 가능한지 확인한다.
7. 사용자가 확정하면 `확정된 요구사항`으로 정리한다.
8. 확정 후에만 작업 분해, 실행 계획, 검증 계획을 작성한다.
9. 사용자가 확정을 거부하거나 수정하면 다시 질문하고 요약을 갱신한다.

## 출력 형식: 확정 전
요구사항이 아직 확정되지 않았으면 아래 구조를 사용한다.

```markdown
## 현재 이해한 요구사항

## 확정된 내용

## 아직 모호한 내용

## 확인 질문
1. ...

## 확정 요청
위 요구사항으로 확정해도 될까요?
```

## 출력 형식: 확정 후
사용자가 요구사항을 확정하면 아래 구조를 사용한다.

```markdown
## 확정된 요구사항

## 제외 범위

## 성공 기준

## 작업 분해
| ID | 작업 | 의존성 | 병렬 가능 | 산출물 |
| --- | --- | --- | --- | --- |

## 실행 계획

## 검증 계획

## 리스크

## 다음 행동
```

## 질문 작성 기준
- 질문은 사용자의 의사결정이 필요한 것만 묻는다.
- 코드, 설정, 문서에서 확인 가능한 사실은 직접 조사하고, 사용자에게 사실 확인을 떠넘기지 않는다.
- 질문은 선택지나 예시를 포함해 답하기 쉽게 만든다.
- 범위, 제외 범위, 성공 기준, 우선순위, 사용자 기대 결과를 우선 확인한다.

## 완료 조건
- 사용자가 요구사항 요약에 동의했다.
- 확정된 요구사항, 제외 범위, 성공 기준이 명확하다.
- 작업 분해와 실행 계획은 확정된 요구사항을 기준으로 작성됐다.
- 남은 모호점이 있다면 진행을 막는 항목과 나중에 정해도 되는 항목이 분리되어 있다.
