# 의료 음성 노트 기능을 내부 AI 구조로 바꾸며 배운 것들

## 시작은 단순해 보였다

처음에는 음성 노트 기능을 단순하게 생각했다.

진료 중 대화를 녹음하고, 녹음 파일을 STT로 변환하고, 그 전사 결과를 바탕으로 경과기록 초안을 만들면 되는 일처럼 보였다. PoC 단계에서는 실제로 그렇게 접근해도 충분했다. 화면에서 녹음 파일을 만들고, 백엔드에서 STT와 LLM을 호출하고, 결과를 다시 화면에 보여주면 됐다.

그런데 운영 구조로 가져가려면 이야기가 달라졌다.

외부 AI API를 직접 호출하던 구조를 내부 AI 게이트웨이 기반으로 바꿔야 했고, 음성 파일도 단순히 API에 바로 넘기는 것이 아니라 내부 파일 저장소와 비동기 처리 흐름을 거쳐야 했다.

결국 이 작업은 “AI API 주소 하나 바꾸기”가 아니었다.

녹음 파일의 생명주기, 내부 인증, 비동기 작업 상태, 결과 정규화, LLM 검증, 화면 검토 흐름까지 같이 다시 보는 작업이었다.

## 만들고 싶었던 기능

목표는 진료 흐름에서 실제로 쓸 수 있는 음성 노트 기능이었다.

- 진료 중 의사와 환자의 대화를 녹음한다.
- 녹음 파일을 STT로 변환한다.
- 전사 전문을 보여준다.
- 발화 단위로 의사/환자 역할을 검토할 수 있게 한다.
- 전문을 기준으로 SOAP 경과기록 초안을 생성한다.
- 사용자가 전문을 수정하면 SOAP만 다시 생성할 수 있게 한다.
- 최종적으로 경과기록 템플릿에 자연스럽게 적용한다.

여기서 중요한 점은 이 기능이 진단이나 처방을 자동으로 확정하는 기능이 아니라는 것이다. 목적은 경과기록 초안 작성 보조다. 최종 확인과 저장 책임은 여전히 의료진에게 있다.

## 실제로 부딪힌 문제

### 1. 음성 파일 하나 보내면 끝나는 구조가 아니었다

처음에는 화면에서 만든 음성 파일을 백엔드로 보내고, 백엔드가 STT API를 호출하면 끝날 줄 알았다.

하지만 내부 AI 처리 구조에서는 음성 파일을 먼저 내부 파일 저장소에 올리고, 그 업로드 결과를 STT 작업 생성 요청에 넘기는 흐름이 필요했다.

```text
화면 녹음 파일
-> 서비스 백엔드
-> 내부 파일 저장소 업로드
-> 내부 AI STT 작업 생성
-> 상태 조회
-> 결과 조회
-> LLM 처리
-> 임시 파일 정리
```

이 구조를 받아들이고 나니 책임 경계를 다시 잡아야 했다.

화면은 녹음과 결과 검토에 집중하고, 백엔드는 내부 파일 저장소와 AI 게이트웨이 호출을 감싸는 adapter 역할을 맡는 쪽이 자연스러웠다.

### 2. 공통 호출 유틸을 쉽게 건드리면 안 됐다

초반에는 기존 공통 API 호출 유틸을 그대로 재사용하거나, 필요한 기능을 조금 추가하면 되지 않을까 생각했다.

그런데 내부 인증 헤더, 파일 업로드 방식, JSON 호출 방식이 섞이면서 영향 범위가 커졌다. 공통 유틸에 기능을 추가하면 음성 노트와 상관없는 다른 API 호출에도 영향을 줄 수 있었다.

그래서 방향을 바꿨다.

- 공통 유틸은 건드리지 않는다.
- 음성 노트 전용 client를 서비스 백엔드 안에 둔다.
- 전용 client는 필요한 내부 API 호출만 얇게 감싼다.
- 인증 헤더나 요청 형식은 해당 기능 내부에서만 통제한다.

이렇게 하니 문제를 추적하기 쉬워졌다. 파일 업로드에서 실패했는지, STT 작업 생성에서 실패했는지, 결과 조회에서 실패했는지 단계별로 로그를 나눌 수 있었다.

### 3. STT 결과가 바로 오지 않았다

가장 중요한 차이는 STT가 동기 결과를 바로 주지 않는다는 점이었다.

작업 생성 API는 최종 전사 결과를 반환하지 않는다. 대신 작업 식별자를 반환하고, 실제 음성 처리와 script 생성은 내부에서 비동기로 진행된다.

그래서 화면은 작업 식별자를 받은 뒤 상태를 계속 확인해야 한다.

```text
작업 시작 요청
-> 작업 식별자 수신
-> 일정 간격으로 상태 조회
-> 완료되면 결과 조회
-> 실패하면 오류 처리
-> 일정 시간 이상 지나면 timeout 처리
```

이 구조를 처음 보면 “그냥 한 번에 기다리면 안 되나?”라는 생각이 든다.

하지만 STT는 파일 크기, 음성 길이, 외부 provider 상태에 따라 처리 시간이 달라진다. HTTP 요청 하나를 계속 붙잡는 방식은 브라우저, gateway, 서버 timeout에 취약하다.

비동기 작업으로 보고 polling하거나 push 기반 상태 수신을 설계하는 편이 더 안전하다.

### 4. 완료 상태를 보자마자 파일을 지우면 안 됐다

음성 파일은 민감할 수 있으므로 임시 파일을 계속 남겨두면 안 된다.

그렇다고 STT 상태가 완료됐다는 이유만으로 바로 삭제하면 또 문제가 생긴다. 상태가 완료됐다는 것은 AI 처리 쪽에서 작업이 끝났다는 의미일 뿐, 우리 서비스가 결과를 조회하고 정규화하고 화면에 내려주는 작업까지 끝났다는 뜻은 아니다.

그래서 삭제 기준을 이렇게 나눴다.

- 작업 시작 전에 실패했다면 업로드된 파일만 정리한다.
- STT가 실패 상태로 끝났다면 임시 파일을 정리한다.
- STT가 완료됐더라도 결과 조회 전에는 삭제하지 않는다.
- 결과 조회와 후속 처리가 끝난 뒤 임시 파일을 정리한다.

이 기준을 잡고 나니 “언제 지워도 되는가”가 명확해졌다.

### 5. 화자 구분과 역할 분류는 다른 문제였다

STT가 speaker label을 줄 수 있다고 해도, 그 label이 곧 “의사”나 “환자”를 의미하지는 않는다.

예를 들어 STT 결과는 이런 식으로 나올 수 있다.

```text
[00:00:00 - 00:00:03] 화자 1
안녕하세요 어디가 불편하세요

[00:00:04 - 00:00:08] 화자 2
목이 아프고 열이 납니다
```

여기서 `화자 1`, `화자 2`는 음성 기반 분리 결과다. 실제 역할은 발화 내용을 보고 판단해야 한다.

그래서 LLM 처리도 두 가지로 나눴다.

- 전체 전문을 기준으로 SOAP 초안을 만든다.
- 발화 단위 텍스트를 기준으로 의사/환자/미정 역할을 분류한다.

이렇게 나누면 STT 결과와 사용자 검토 UI가 더 자연스럽게 이어진다.

## 어떻게 해결했나

### 기능 기준으로 adapter를 분리했다

이 기능은 경과기록과 연결되지만, 사용자 입장에서의 기능 이름은 음성 노트다.

그래서 백엔드 안에서도 경과기록 하위 로직에 모든 것을 밀어 넣기보다, 음성 노트 전용 영역을 만들고 그 안에서 내부 AI 연동을 담당하게 했다.

역할은 크게 네 가지로 나눴다.

- controller: 화면에서 호출하는 endpoint 제공
- service: STT 시작, 상태 조회, 결과 조회, LLM 처리, cleanup 순서 제어
- api client: 내부 파일 저장소와 AI 게이트웨이 호출
- prompt manager: SOAP 생성과 화자 역할 분류 prompt 관리

화면은 이 내부 구조를 몰라도 된다. 화면은 음성 파일을 보내고, 상태를 확인하고, 결과를 표시하면 된다.

### STT와 LLM 실패를 분리했다

처음에는 STT 결과만 보여줘도 테스트가 가능했다.

하지만 실제 기능으로 보면 전사 전문과 SOAP 초안이 모두 필요하다. 또한 사용자가 전문을 수정했을 때 STT를 다시 돌릴 필요는 없다.

그래서 흐름을 이렇게 나눴다.

```text
음성 파일
-> STT
-> 전문/발화 목록
-> SOAP 생성
-> 화면 검토
-> 전문 수정
-> SOAP만 재생성
```

이 구조의 장점은 실패를 분리할 수 있다는 점이다.

STT가 성공했는데 SOAP 생성만 실패했다면 녹음 파일을 다시 보낼 필요가 없다. 반대로 STT 자체가 실패했다면 사용자에게 녹음 파일 재전송이나 재시도 흐름을 제공해야 한다.

### prompt는 목적별로 나눴다

LLM에게 한 번에 모든 것을 시키는 대신 목적을 나눴다.

첫 번째 prompt는 SOAP 경과기록 초안을 만든다.

```json
{
  "subjective": "",
  "objective": "",
  "assessment": "",
  "plan": ""
}
```

두 번째 prompt는 발화별 역할을 분류한다.

```json
{
  "segmentRoles": [
    {
      "order": 1,
      "speaker": "화자 1",
      "role": "의사"
    }
  ]
}
```

LLM 응답은 항상 검증이 필요하다. 구조화 출력을 요청하더라도 실제 응답이 원하는 JSON 형태로 온다는 보장은 별개의 문제다. 그래서 백엔드에서는 JSON 파싱, 필드 존재 여부, 빈 값 처리 기준을 가져야 한다.

### UI는 검증에 필요한 만큼만 바꿨다

이번 작업의 본질은 UI 개편이 아니었다.

그래서 기존 음성 플레이어의 큰 모양은 유지하고, 검증에 필요한 기능만 붙였다.

- 기존 녹음 파일로 다시 테스트할 수 있는 파일 테스트 버튼
- STT 전체 처리 시간 console 출력
- 좌측 전문 / 우측 SOAP 배치
- 발화별 역할 선택
- 전문 수정 후 SOAP 재생성
- 경과기록 템플릿에 SOAP 구조 적용

특히 파일 테스트 버튼은 효과가 컸다. 매번 다시 녹음하지 않고 같은 파일로 STT와 LLM 흐름을 반복 검증할 수 있었기 때문이다.

## 지금 구조에서 얻은 것

첫 번째로, 외부 AI 직접 호출을 내부 AI 게이트웨이 구조로 옮기면서도 사용자 흐름은 유지할 수 있었다.

두 번째로, STT와 LLM 실패를 분리해서 볼 수 있게 됐다. 이건 재시도 UX를 설계할 때 중요하다.

세 번째로, 임시 음성 파일을 언제 삭제해야 하는지 기준이 생겼다. 보안과 복구성 사이에서 어느 시점까지 파일을 남길지 판단할 수 있게 됐다.

네 번째로, 화면과 백엔드의 책임이 조금 더 분명해졌다. 화면은 녹음과 검토, 백엔드는 내부 연동과 상태 제어에 집중한다.

## 아직 고민 중인 것

### polling으로 인한 체감 지연

현재는 일정 간격으로 상태를 조회한다.

구현과 테스트는 쉽지만, 상태가 완료된 직후에도 다음 polling 시점까지 화면은 완료를 모른다. 짧은 녹음에서는 이 대기 시간이 꽤 크게 느껴질 수 있다.

해결 후보는 몇 가지가 있다.

- 초반에는 polling 간격을 짧게 두고, 시간이 길어질수록 간격을 늘린다.
- push 기반 상태 수신을 붙이고 polling은 fallback으로 둔다.
- 백엔드가 짧은 시간만 대기해보고, 오래 걸리면 비동기 상태 조회로 전환한다.
- 전체 처리 시간을 단계별로 측정해 병목이 polling인지, STT 처리 자체인지 구분한다.

개인적으로는 push 방식을 바로 기본값으로 두기보다 fallback polling까지 같이 설계하는 편이 맞다고 본다. 운영 환경에서는 구독 타이밍을 놓치거나 연결이 끊길 수 있기 때문이다.

### 실패 후 재시도와 파일 보관

지금 구조는 임시 파일을 빨리 정리하는 쪽에 가깝다.

이 방식은 파일 누적을 막는 장점이 있지만, 실패 직후 같은 음성 파일로 재시도하기는 어렵다.

앞으로는 정책을 정해야 한다.

- 화면이 마지막 녹음 파일을 세션 안에서 잠시 들고 있을 것인가?
- 백엔드가 임시 파일을 일정 시간 보관할 것인가?
- STT 실패와 LLM 실패를 어떻게 다르게 재시도할 것인가?
- 사용자가 보는 오류와 개발자가 보는 실패 원인을 어디까지 분리할 것인가?

특히 외부 또는 내부 AI provider가 실패 사유를 충분히 내려주지 않으면 디버깅이 어려워진다. 이 경우 서비스 쪽 로그만으로는 한계가 있으므로 provider 응답 계약 자체를 보완해야 한다.

### audio seek

전문에서 특정 발화를 클릭하면 그 시점부터 녹음을 재생하는 기능은 꽤 좋아 보인다.

```text
발화 클릭
-> segment.start 확인
-> audio.currentTime 이동
-> 해당 구간 재생
```

다만 이 기능은 segment timestamp가 안정적으로 있어야 한다. STT 결과의 시간 정보 품질을 먼저 확인한 뒤 붙이는 편이 맞다.

### 사용량과 비용 추적

STT와 LLM을 모두 내부 AI 게이트웨이로 호출하면 사용량과 비용 추적도 같이 봐야 한다.

단순히 “호출이 된다”가 끝이 아니다.

- 테스트 환경에서 차단 없이 호출되는가?
- 실제 사용량은 어디에 기록되는가?
- STT와 LLM이 각각 어떻게 집계되는가?
- 실패한 요청도 사용량에 포함되는가?
- 서버 간 호출에서 사용자 식별이 정확히 이어지는가?

이 부분은 기능 품질과 별개로 운영 전 반드시 확인해야 한다.

## 정리하면

이번 작업은 “AI API 하나 교체”가 아니었다.

음성 노트라는 사용자 기능을 기준으로 보면, 녹음 파일의 생명주기, 내부 인증, 비동기 STT 상태, LLM 결과 검증, UI 검토 흐름, cleanup 정책까지 같이 묶여 있었다.

그래서 방향은 이렇게 잡았다.

- 화면은 음성 노트 경험에 집중한다.
- 백엔드는 내부 파일 저장소와 AI 게이트웨이를 감싸는 adapter 역할을 담당한다.
- STT는 비동기 작업으로 보고 상태 조회 흐름을 설계한다.
- LLM은 SOAP 생성과 화자 역할 분류를 분리한다.
- 임시 음성 파일은 결과 확보 후 정리한다.
- 운영 전에는 polling 지연, push fallback, 실패 재시도, audio seek, 사용량 정책을 추가로 본다.

결국 중요한 건 “일단 돌아가게 만들기”가 아니었다.

어디가 책임지고, 실패했을 때 어디를 봐야 하며, 사용자가 다시 시도할 수 있는 단위가 무엇인지 분리하는 일이었다.

이 기준이 잡혀야 다음 기능을 붙일 때도 덜 흔들린다.
