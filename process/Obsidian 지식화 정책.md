# Obsidian 지식화 정책

## 왜 필요한가

작업 내용을 시간순 로그로 남기면 나중에 다시 참조하기 어렵다.

사용자가 원하는 문서화는 “무엇을 했는가”의 기록이 아니라, 수정 내용을 분석해 다음 작업에서 재사용할 수 있는 기술 문서, 프로세스 문서, 의사결정 문서, 실패 기록, 커리어 성장 서사로 정리하는 것이다.

## 작성 말투

문서는 사용자가 말하듯이 작성한다.

핵심은 “내가 어떤 문제를 봤고, 왜 그렇게 판단했고, 어떻게 해결했고, 어떤 기대효과가 있었는지”를 기술 블로그처럼 설명하는 것이다.

문장은 과장하지 않고 실무적으로 쓴다. 기술적 판단과 비즈니스/운영 효과를 분리해서 설명한다.

## 어떻게 실행하는가

작업 완료 후 먼저 문서 유형을 분류한다.

- 기술 구조, 코드 흐름, API 계약, 상태 관리, 테스트 전략은 `technical-notes`, `architecture`, `patterns`에 정리한다.
- 작업 방식, 리뷰 기준, 운영 절차, 문서화 규칙은 `process`, `workflows`에 정리한다.
- 장기적으로 영향을 주는 선택은 `decisions`에 정리한다.
- 재발 가능한 실수나 디버깅 난이도가 높은 문제는 `failures`에 정리한다.
- 기술적 또는 비즈니스적으로 좋은 개선은 `career/성과와 기술 성장 포트폴리오.md`에 요약하여 누적한다.

새 문서를 만들기 전에 기존 문서에 병합할 수 있는지 먼저 확인한다.

`career` 문서는 상세 분석 문서가 아니라 요약 인덱스다.

상세한 기술 설명은 `technical-notes`, `failures`, `process`, `decisions`에 작성하고, `career/성과와 기술 성장 포트폴리오.md`에는 핵심 요약과 상세 문서 링크만 남긴다.

## 제목과 파일명

문서 제목과 파일명은 날짜나 작업명보다 큰 주제를 기준으로 작성한다.

예를 들어 `2026-05-18-patient-info-widget-dirty-gate.md`보다 `EMR 포틀릿 환자 변경 안정성.md`가 좋다.

날짜는 파일명이 아니라 frontmatter의 `date`에 기록한다.

파일명과 제목은 되도록 한글로 작성하고, 코드 식별자나 컴포넌트명처럼 원문이 더 명확한 경우에만 영어를 섞는다.

## 문서 작성 기준

모든 문서는 한국어로 작성한다.

각 문서는 최소한 다음 질문에 답해야 한다.

- 왜 필요한가?
- 왜 이 기술 또는 접근 방식을 선택했는가?
- 어떻게 해결했는가?
- 어떤 기대효과가 있는가?
- 기술적으로 어떤 가치가 있는가?
- 비즈니스 또는 운영 관점에서 어떤 가치가 있는가?
- 다음에 어떻게 재사용할 수 있는가?
- 남은 리스크는 무엇인가?

## 시각화 기준

기술 흐름이나 프로세스를 설명할 때는 다이어그램을 적극 활용한다.

- 컴포넌트 호출 흐름은 Mermaid `sequenceDiagram`
- 상태 변경이나 조건 분기는 Mermaid `flowchart`
- 구조 설명은 Mermaid `graph`
- 다이어그램 뒤에는 핵심 해석을 한글로 덧붙인다.

## 기대효과

Obsidian Vault가 단순 작업 이력이 아니라, 반복 가능한 기술 판단과 프로세스 개선의 저장소가 된다.

다음 세션에서 유사한 문제를 만났을 때 기존 문서를 바로 참조할 수 있고, 장기적으로는 자기소개, 면접, 기술 이력서에 반영할 수 있는 성과와 성장 서사가 누락되지 않는다.

## 운영/협업 가치

작업 결과가 개인 기억에 의존하지 않고 문서로 축적된다.

리뷰, 리팩토링, 디버깅, 프로세스 개선의 이유와 기대효과가 남기 때문에, 나중에 같은 결정을 반복 검토하는 비용을 줄일 수 있다.

## 체크리스트

- Vault root가 확정되어 있는가?
- 기존 문서에 병합할 수 있는가?
- 작업 로그가 아니라 지식 문서로 작성했는가?
- 제목과 파일명이 큰 주제 기반인가?
- 파일명과 제목을 되도록 한글로 작성했는가?
- 왜/어떻게/기대효과를 포함했는가?
- 기술 블로그처럼 문제 배경과 흐름을 설명했는가?
- 필요한 경우 다이어그램을 포함했는가?
- 기술 선택 이유와 대안 검토가 필요한 경우 이를 기록했는가?
- 기술적 가치와 비즈니스 가치를 분리했는가?
- 다른 사람에게 설명 가능한 성과/성장 서사가 있으면 `career/성과와 기술 성장 포트폴리오.md`에 요약과 상세 문서 링크를 누적했는가?

## 예외 상황

단순 질의응답, 일회성 확인, 의미 있는 기술 판단이 없는 작은 수정은 문서를 만들지 않을 수 있다.

단, 그 과정에서 재발 가능한 실수나 프로세스 개선점이 발견되면 별도 문서화한다.
