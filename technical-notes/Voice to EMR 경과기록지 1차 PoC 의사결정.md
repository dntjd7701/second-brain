# Voice to EMR 경과기록지 1차 PoC 의사결정

## 시작점

Voice to EMR 기능을 처음 떠올렸을 때 가장 큰 유혹은 한 번에 크게 가는 것이었다. 의사와 환자의 대화를 녹음하고, 그 안에서 SOAP를 만들고, 나아가 처방과 상병까지 자동 입력하는 그림은 제품적으로 매력적이다. 하지만 실제 구현을 시작하기 전에 먼저 잘라야 할 범위가 있었다.

이번 1차 목표는 처방이나 상병이 아니다. 경과기록지 작성 보조만 본다.

즉 목표는 다음 문장으로 고정한다.

> 녹음기 버튼을 활성화한 뒤 의사와 환자의 대화를 전사하고, 그 내용을 SOAP 기준으로 정리하여 경과기록지 draft에 입력한다. 최종 저장은 사용자가 검토 후 기존 경과기록 저장 버튼으로 수행한다.

이 정의가 중요한 이유는 AI가 의료 판단과 저장 책임을 가져가지 않게 만들기 위해서다. 1차 PoC의 산출물은 확정 기록이 아니라 초안이다. 기존 경과기록지 저장/조회 흐름은 [[경과기록지_조회와_신규작성_분리_기준]]에서 정리한 기준을 유지한다.

## 문제를 어떻게 지정했는가

처음 문제는 "음성으로 경과기록을 써준다"처럼 보였다. 하지만 이 표현은 너무 넓다. 실제로는 적어도 네 가지 문제가 섞여 있다.

- 마이크를 켜고 대화를 수집하는 문제
- 음성을 텍스트로 바꾸는 STT 문제
- 대화 텍스트를 SOAP 기준으로 정리하는 문제
- 정리된 내용을 기존 경과기록지에 안전하게 입력하고 저장하는 문제

이 중 1차 PoC에서 가장 먼저 검증해야 하는 것은 마지막 문제다. 경과기록지 화면에 어떤 방식으로 초안을 넣어야 자연스러운지, 기존 dirty/save 흐름을 깨지 않는지, 사용자가 수정 후 저장하는 흐름이 맞는지 확인해야 한다.

따라서 이번 문제 정의는 "최고 품질 STT/LLM을 붙이는 것"이 아니라 "Voice to EMR 업무 흐름이 경과기록 프로세스에 자연스럽게 들어갈 수 있는지 확인하는 것"이다.

## 버린 선택지

### OpenAI/Azure를 즉시 붙이는 방식

OpenAI나 Azure OpenAI를 바로 붙이면 STT와 SOAP 정규화 품질을 빠르게 확인할 수 있다. 하지만 1차 기능 구현 단계에서는 비용, API key 관리, 환자 대화 외부 전송 여부, 보안 심사, 모델 선택 같은 결정이 먼저 끼어든다.

지금 필요한 것은 운영 모델 결정이 아니라 경과기록지 안에서 사용자가 어떤 흐름으로 초안을 받아들이는지 확인하는 것이다. 그래서 유료 AI 연동은 1차에서 제외한다.

### AI가 경과기록을 바로 저장하는 방식

이 방식도 제외한다. 경과기록은 의료 문서이므로 AI 결과를 곧바로 DB에 저장하면 책임 경계가 모호해진다. 1차에서는 AI 또는 음성인식 결과가 "draft 입력"까지만 담당한다.

저장은 사용자가 확인하고 기존 경과기록 저장 프로세스를 통해 수행한다.

### 처방/상병까지 함께 추출하는 방식

처방과 상병은 경과기록보다 훨씬 강한 검증과 코드 매칭이 필요하다. 처방은 처방 그리드, 급여/비급여, DUR, 권고/불가, 진료과/환자구분 조건과 연결된다. 상병도 코드피커와 주상병/부상병 판단이 필요하다.

따라서 1차에서는 처방/상병 후보 추출을 의도적으로 제외한다. 나중에 고도화할 때 별도의 후보 추천 흐름으로 분리한다.

## 초기 판단: Web Speech API를 포함할 수 있다고 본 이유

PoC 목적에서는 Web Speech API를 포함할 수 있다. 이유는 단순하다. 비용 없이 브라우저 수준에서 음성 인식 흐름을 붙여볼 수 있기 때문이다.

단, Web Speech API를 운영 품질의 STT 엔진으로 간주하지 않는다. 브라우저와 OS에 따라 동작 차이가 있고, 일부 환경에서는 서버 기반 인식 엔진을 사용할 수 있다. 의료 용어 정확도, 화자 구분, 감사 가능성, 보안 정책도 별도로 검토해야 한다.

초기에는 1차 PoC에 충분하다고 보았다. 이 단계의 목적은 "진료실 대화가 어느 정도로 전사되는가"보다 "전사된 텍스트가 경과기록지 초안으로 들어가는 사용자 경험이 맞는가"를 확인하는 데 있었기 때문이다. 다만 실제 테스트 뒤에는 아래 재정의에 따라 브라우저 STT 미리보기를 제거했다.

## 2026-06-05 재정의: 브라우저 STT 미리보기 제거

OpenAI STT를 붙인 뒤 다시 문제를 나누어 보니, 1차 PoC에서 필요한 "실시간성"은 실제로 두 종류였다.

- 사용자가 녹음 중이라는 것을 체감하는 UI 실시간성
- 경과기록 초안에 사용할 최종 전사의 정확도

처음에는 이 둘을 분리해, 녹음 중에는 Web Speech API로 실시간 미리보기를 보여주고 최종 전사는 녹음 blob 기반 OpenAI STT로 처리하는 방식을 검토했다. 이 접근은 비용이 들지 않고 화면이 즉시 반응한다는 장점이 있었다. 하지만 실제 테스트에서 브라우저 STT는 다음 문제를 드러냈다.

- AirPods 등 마이크 입력이 잡혀도 `no-speech` 또는 `onend`가 반복되어 세션이 끊긴다.
- 미리보기 텍스트가 짧은 발화, 잡음, 한국어 의학 표현에서 불안정하게 표시된다.
- 화면에 보이는 텍스트와 최종 전사 텍스트가 서로 다르면 사용자가 어느 값을 신뢰해야 하는지 혼란스럽다.
- 브라우저/OS/권한 상태에 따라 동작이 달라 운영 재현성과 감사 가능성이 낮다.
- 미리보기라도 의료 문서 후보처럼 보이면, 부정확한 문장이 신뢰를 떨어뜨린다.

따라서 1차 PoC에서는 브라우저 STT 미리보기를 제거한다. 녹음 중 실시간성은 텍스트가 아니라 파형, 타이머, live 상태, 로딩 상태로 표현한다. 경과기록 초안의 기준 데이터는 오직 녹음 종료 후 확정된 audio blob을 STT provider에 전달해 받은 최종 전사 결과다.

새 기준선은 다음과 같다.

```text
녹음 중:
MediaRecorder로 audio blob을 생성하면서 파형, timer, live 상태를 보여준다.

녹음 종료:
실제 녹음 blob을 OpenAI STT로 전송한다.

최종 전사:
녹음본 기반 OpenAI STT 결과만 전사 내용에 반영한다.

후속 처리:
사용자가 최종 전사 내용을 수정한 뒤 SOAP 정리 또는 원문 적용을 선택한다.
```

이 구조의 핵심은 "보이는 반응"과 "기록 후보 데이터"를 섞지 않는 것이다. 사용자에게 녹음 중임을 충분히 보여주되, 불안정한 브라우저 STT 텍스트는 화면에도 남기지 않는다. 의료 문서 후보로 쓰는 텍스트는 녹음 종료 후 audio blob을 기반으로 생성한 AI STT 결과뿐이다.

UI에는 이 책임 경계를 분명하게 드러내야 한다.

- 녹음 중 표시: `녹음 중`, `00:00:00`, 파형 animation
- 녹음 종료 후 로딩: `녹음본 기준 최종 전사 생성 중`
- 전사 결과 영역: `최종 전사 결과`
- 안내 문구: `전사 결과는 녹음 종료 후 생성됩니다.`

이 문구가 필요한 이유는 사용자가 녹음 중 화면 반응을 최종 의료기록 후보로 착각하지 않게 하기 위해서다.

## 비용과 성능 비교

2026-06-05 기준 OpenAI 공식 가격표를 보면, 같은 "음성을 텍스트로 바꾼다"는 목표라도 선택한 방식에 따라 비용과 운영 난이도가 크게 달라진다.

| 방식 | 용도 | 비용 기준 | 장점 | 단점 | 1차 PoC 판단 |
|---|---|---:|---|---|---|
| 브라우저 STT | 녹음 중 실시간 미리보기 | 무료 | 즉각적 UI 반응, 구현 단순 | 정확도 낮음, 브라우저/OS 의존, 최종 전사와 혼선 | 1차 기준선에서 제거 |
| `gpt-4o-mini-transcribe` | 녹음 종료 후 blob 최종 전사 | 약 `$0.003/min` | 저렴, 현재 REST 구조와 맞음 | 실시간 전사는 아님 | 1차 최종 전사 기준으로 적합 |
| `gpt-4o-transcribe` | 더 높은 품질의 blob 최종 전사 | 약 `$0.006/min` | mini보다 품질 기대 | mini 대비 비용 2배 | 품질 검증 단계 후보 |
| `gpt-realtime-whisper` | 진짜 AI 실시간 STT | 약 `$0.017/min` | 낮은 지연의 transcript delta | WebSocket 운영 필요, mini blob 대비 약 5.7배 | 2차 확장 후보 |
| `gpt-realtime` | 실시간 음성 agent | audio/text token 과금 | 음성 입출력, 도구 호출, 대화형 판단 | 단순 STT에는 과함 | 현 단계 부적합 |

5분 진료 1,000건을 단순 계산하면 다음과 같다. SOAP LLM 비용은 별도이며, 아래는 STT만 비교한 것이다.

```text
gpt-4o-mini-transcribe:
5분 * 1,000건 * $0.003/min = 약 $15

gpt-4o-transcribe:
5분 * 1,000건 * $0.006/min = 약 $30

gpt-realtime-whisper:
5분 * 1,000건 * $0.017/min = 약 $85
```

따라서 현재 PoC의 비용/효과 기준은 다음과 같이 정리한다.

> 1차 PoC는 녹음 중 UI 반응성을 파형/타이머로 확보하고, 최종 전사는 녹음 blob 기반 OpenAI STT로 처리한다.

이 접근은 사용자에게 "녹음되고 있다"는 감각을 주면서도, 최종 의료 문서 후보는 녹음본 기반 전사 결과만 사용한다. Realtime AI STT는 실시간 확정 전사, 화자 분리, 대화 중 SOAP 후보 생성이 실제 요구사항으로 확정될 때 별도 확장 단계에서 검토한다.

## 1차 결론

1차 PoC의 핵심 결론은 다음과 같다.

> AI 없이도 경과기록 Voice to EMR의 업무 흐름은 구현 가능하다. 다만 운영 품질 검증 단계에서는 STT provider와 SOAP normalizer를 교체 가능한 구조로 분리해야 한다.

이때 1차의 성공 기준은 품질이 아니라 설득력이다. SOAP 자동 분류가 완벽한지보다, 경과기록지 안에서 녹음 버튼을 누르고, 녹음 중임을 명확히 보고, 변환 중 로딩을 거쳐, 결과가 자연스럽게 기록지에 들어가는 장면을 보여줄 수 있어야 한다.

구현은 다음처럼 접근한다.

```text
녹음 버튼
-> MediaRecorder 녹음
-> OpenAI STT 최종 전사
-> LLM SOAP 정리
-> 경과기록 draft 삽입
-> 사용자 검토
-> 기존 저장 버튼으로 저장
```

이 구조에서 MediaRecorder는 음성 파일 수집 수단이고, STT provider와 SOAP normalizer는 나중에 교체 가능한 지점으로 남긴다. 브라우저 STT와 키워드 mock SOAP은 의료기록 후보처럼 오해될 수 있어 기준선에서 제거한다.

```text
MediaRecorderAiSttProvider -> OpenAiSttProvider / AzureSttProvider / InternalSttProvider
LlmSoapNormalizer -> 다른 LLM provider / 내부 SOAP normalizer
```

2026-06-03 1차 구현에서는 Voice to EMR 결과를 기존 draft에 덮어쓰지 않고 새 경과기록 draft sheet로 추가하는 방식을 선택했다. 이유는 시연 안정성이다. 사용자가 이미 작성하던 경과기록을 잃지 않고, 음성으로 생성된 초안이 경과기록지 안에 별도 draft로 들어오는 장면을 명확히 보여줄 수 있다. 기존 draft SOAP 영역 교체나 커서 위치 삽입은 이후 실제 사용성 검증 단계에서 다시 판단한다.

## AI가 필요한 시점

AI는 1차 기능 구현에 필수는 아니다. 하지만 운영 품질을 검증하려면 필요해질 가능성이 높다.

그 이유는 다음과 같다.

- 진료실 대화에는 잡음, 끼어들기, 말 흐림이 많다.
- 의학 용어, 약어, 증상 표현은 일반 STT에서 오인식될 수 있다.
- SOAP 정리는 단순 키워드 분류가 아니라 맥락 이해가 필요하다.
- 대화에 없는 내용을 만들어내지 않는 안전 규칙이 필요하다.
- 원문 전사, 요약 결과, 경고, 신뢰도를 분리해서 추적해야 한다.

따라서 AI는 처음부터 붙이는 대상이 아니라, PoC로 업무 흐름을 검증한 뒤 품질 엔진으로 교체할 대상이다.

## 발표용 메시지

이 의사결정의 발표용 메시지는 "AI를 붙였다"가 아니다.

더 중요한 메시지는 다음이다.

> 먼저 경과기록 작성 업무 흐름을 검증하고, 이후 STT/LLM 품질 엔진을 교체 가능하게 설계했다.

이 접근은 비용과 보안 리스크를 낮춘다. 또한 실제 사용자가 경과기록 초안을 어떻게 받아들이고 수정하는지 먼저 볼 수 있다. 큰 AI 도입보다 작은 업무 흐름 검증이 먼저다.

시연에서는 UI의 이미지가 중요하다. 경과기록 컴포넌트의 `기록지 추가`가 있는 영역에 녹음기 버튼을 두어 사용자가 "기록지를 추가하거나 음성으로 초안을 생성한다"는 맥락을 바로 이해하게 한다. 녹음 중에는 버튼 상태, 색상, 진행 표시, "녹음 중" 문구가 분명해야 한다. 녹음 종료 후에는 전사/변환/등록 단계가 로딩 상태로 이어지고, 사용자는 결과가 갑자기 튀어나오는 것이 아니라 처리 과정을 따라가며 경과기록 draft에 입력되는 장면을 보게 된다.

따라서 1차 PoC는 다음을 설득해야 한다.

- 이 기능은 경과기록 작성 흐름 안에 자연스럽게 배치될 수 있다.
- 사용자는 녹음 중, 변환 중, 적용 완료 상태를 직관적으로 이해할 수 있다.
- SOAP 품질 엔진은 나중에 교체하더라도 UI와 저장 흐름은 이미 성립한다.
- 최종 저장은 여전히 사용자가 검토 후 수행하므로 의료 문서 책임 경계가 유지된다.

## 다음 재검토 조건

다음 조건 중 하나가 충족되면 유료 또는 내부 STT/LLM provider 검토로 넘어간다.

- 녹음 blob 기반 STT 전사 품질이 진료실 테스트에서 부족하다고 확인된다.
- 의학 용어 오인식이 경과기록 품질에 영향을 준다.
- 화자 구분이 SOAP 정리 품질에 결정적으로 필요하다.
- 보안 정책상 브라우저 음성 인식 처리 경로를 통제해야 한다.
- 실제 운영 규모의 비용/성능/감사 로그 기준을 산정해야 한다.

이때도 원칙은 유지한다. AI는 저장자가 아니라 초안 작성 보조자다.

## 2026-06-05 제품 흐름 재정의: 전문보다 저장 경험 우선

구현을 진행하면서 화면 흐름을 다시 판단했다. 초기에는 녹음 종료 후 전사 전문과 화자 구분을 먼저 보여주고, 사용자가 SOAP 정리 또는 원문 적용을 선택하는 구조였다. 하지만 실제 사용자 관점에서는 매번 전문을 먼저 볼 필요가 없다고 판단했다.

의사는 경과기록을 작성하기 위해 Voice Note를 쓰는 것이지, STT 전문을 검수하기 위해 기능을 여는 것이 아니다. 전문은 문제가 있을 때 확인하는 보조 정보에 가깝다. 따라서 1차 제품 흐름은 다음처럼 바꾼다.

```text
MEDA0010 메인 Voice Note 버튼
-> 우측 하단 플로팅 플레이어
-> 다른 화면을 보거나 조작하면서 녹음
-> 정지 시 플레이어에서 처리 로딩
-> 녹음 blob 기반 STT
-> SOAP 초안 자동 생성
-> Voice Note 결과 다이얼로그에는 SOAP 초안만 우선 표시
-> 저장 버튼으로 기존 ProgressNote 저장 흐름 실행
```

전사 전문과 화자 구분은 기본 화면에서 숨긴다. 사용자가 `전문 보기`를 누를 때만 기존 STT 결과를 기반으로 speaker role 후처리를 호출하고, 다이얼로그를 확장해 전문과 화자 라벨을 보여준다.

이 결정을 한 이유는 세 가지다.

- 사용자가 즉시 원하는 것은 전문 검토가 아니라 경과기록 초안 확인이다.
- 녹음은 진료 화면을 보면서 진행되어야 하므로, 중앙 모달보다 우측 하단 플로팅 플레이어가 더 자연스럽다.
- 사용자의 편의성을 극대화하려면 SOAP 초안 확인 후 바로 저장할 수 있어야 한다.

여기서 "바로 저장"은 AI가 DB에 직접 저장한다는 뜻이 아니다. Voice Note의 저장 버튼은 기존 ProgressNote의 draft 생성과 기존 `handleSave` 흐름을 호출한다. 즉 저장 API, dirty data 구성, 전자인증/권한/기존 저장 규칙은 그대로 유지한다.

```text
Voice Note 저장
-> ProgressNote.handleSaveVoiceEmrNote(noteHtml)
-> RecordSheetService.addSheet(defaultContentHtml)
-> ProgressNote.handleSave()
-> RecordSheetService.insert()
-> /medical/recordSheet/0ho01001
```

이 구조는 사용자 경험을 단순하게 만들면서도 책임 경계를 유지한다. 사용자는 Voice Note 결과 화면에서 바로 저장할 수 있지만, 실제 저장은 기존 경과기록 저장 프로세스를 재사용한다.

## 2026-06-05 전문 지연 조회 결정

전문 보기와 화자 구분은 후순위 기능으로 둔다. 이유는 다음과 같다.

- SOAP 초안이 충분히 납득 가능하면 전문을 볼 필요가 없다.
- 전문과 화자 라벨을 처음부터 보여주면 사용자가 STT 검수 화면으로 오해할 수 있다.
- 화자 역할 분류는 비용과 지연을 추가하므로, 사용자가 요청한 경우에만 수행하는 편이 합리적이다.
- 음성 기록의 목적은 대화 로그 열람이 아니라 경과기록 작성 보조다.

따라서 backend endpoint도 분리한다.

```text
기본 처리:
/medical/recordSheet/voice-to-emr/stt/plain
-> STT-only

전문 보기:
/medical/recordSheet/voice-to-emr/speaker-role
-> speaker segment 기반 role mapping

SOAP 초안:
/medical/recordSheet/voice-to-emr/soap
-> transcript 기반 SOAP JSON
```

이 분리는 비용과 성능을 모두 설명할 수 있는 근거가 된다. 모든 사용자가 매번 전문을 열지 않는다면, 화자 역할 후처리는 필요한 경우에만 호출하면 된다.
