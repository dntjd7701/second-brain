# 경과기록지 조회와 신규작성 분리 기준

## 배경

경과기록지 화면은 서식연동관리의 현재 진료과 경과기록지 서식을 조회해 신규 작성용 템플릿으로 사용한다. 기존 구현처럼 템플릿 조회 결과가 없다는 이유로 전체 그리드 조회를 막으면, 이미 작성된 기록까지 보이지 않는 문제가 생긴다.

## 기준

- 조회 가능 여부와 신규 작성 가능 여부를 분리한다.
- 서식연동관리 미등록은 신규 작성 제한 사유이지, 기존 기록 조회 실패 사유가 아니다.
- 현재 진료과 서식이 조회되면 `mdfrSqs` 기준으로 현재 진료과 서식 문서를 좁혀 조회한다.
- 현재 진료과 서식이 없으면 `mdfrClsfCd` 기준으로 fallback 조회하되, `타 과 보기`가 꺼진 상태에서는 현재 진료/접수 조건으로 좁힌다.
- `타 과 보기`가 켜진 상태에서는 환자/기간/서식연동코드 기준의 타 진료과 기록까지 조회한다.

## 구현 패턴

- FE는 템플릿 조회 실패와 템플릿 미등록을 다르게 처리한다.
- 템플릿 조회 API 자체가 실패하면 사용자에게 오류를 노출한다.
- 템플릿 미등록이면 `recordSheetTemplate`을 `null`로 두고 그리드 조회는 계속 수행한다.
- 신규 기록지 추가 또는 신규 row 저장 시점에만 “서식연동관리에 내 과 경과기록지 서식이 등록되어 있지 않아 신규 작성할 수 없습니다.” 메시지를 노출한다.
- 기존 저장 row 조회는 현재 진료과 템플릿 HTML을 강제로 사용하지 않고, row의 `mdfrSq`로 해당 서식을 다시 조회해 읽는다.

## 재사용 기준

서식연동관리, 병원환경설정, 권한설정처럼 “설정값이 없을 수 있는 기능”은 설정값 부재를 곧바로 조회 실패로 처리하지 않는다. 설정값이 필요한 동작이 무엇인지 먼저 분리하고, 필수 동작에만 명시적으로 차단 메시지를 붙인다.
