# 작성 스타일과 말투

## 목적

Obsidian 문서는 사용자가 직접 말하고 설명하는 방식에 가깝게 작성한다.

이 문서는 앞으로 기술 문서, 프로세스 문서, 커리어 성장 서사를 작성할 때 참고하는 말투와 구조 기준이다.

## 말투

기본은 한국어다.

문장은 실무적으로 쓴다. 과장된 표현보다 “왜 그렇게 판단했는지”, “어떻게 해결했는지”, “무엇이 좋아지는지”가 드러나는 문장을 선호한다.

작업 결과를 단순히 나열하지 않고, 내가 문제를 어떻게 바라봤고 어떤 기준으로 선택했는지를 설명한다.

## 선호하는 질문 구조

- 왜 이게 문제였는가?
- 왜 이 기술 또는 접근을 선택했는가?
- 다른 방법은 왜 제외했는가?
- 어떻게 해결했는가?
- 어떤 기대효과가 있는가?
- 기술적으로 무엇이 좋아졌는가?
- 비즈니스나 운영 관점에서 어떤 의미가 있는가?
- 이 경험을 통해 어떤 성장이나 강점이 드러나는가?

## 기술 블로그 스타일

기술 문서는 기술 블로그처럼 작성한다.

권장 흐름:

```text
문제 제기 → 배경 설명 → 구조 분석 → 해결 접근 → 구현/적용 방식 → 기대효과 → 트레이드오프 → 재사용 포인트
```

코드 변경 자체보다 문제 구조와 판단 과정을 먼저 설명한다.

## 시각화 선호

흐름이 있는 내용은 다이어그램을 적극 활용한다.

Mermaid를 우선 사용한다.

- 컴포넌트 호출 흐름: `sequenceDiagram`
- 상태 변경 흐름: `flowchart`
- 아키텍처 구조: `graph`
- 판단 기준: `flowchart`

다이어그램 뒤에는 반드시 “이 그림에서 중요한 지점”을 설명한다.

## 커리어 서사 관점

목표는 바로 이력서를 쓰는 것이 아니다.

먼저 내가 어떤 일을 했고, 어떤 방식으로 문제를 해결했고, 어떤 기술적 성장을 했는지 설명 가능한 재료를 축적한다.

이후 이 자료를 바탕으로 이력서, 자기소개, 면접 답변, 성과 리뷰에 맞게 가공한다.
